<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| RedisStore → ioredis client → network | every op crosses the network to Redis |
| client key → namespaced KEYS[1] | client-controlled segment enters the Redis keyspace |
| Redis outage → policy | a degraded store decides admit/deny |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-08 | Injection | namespaced Redis key | mitigate | client key passes ONLY as `KEYS[1]` to the defineCommand'd script (parametrized, `numberOfKeys:1`); the prefix is validated (`assertPrefix`) and never client-controlled — no key is concatenated into the Lua body |
| T-02-09 | Denial of Service | hung/slow Redis | mitigate | `commandTimeout` (75ms) bounds every call (DEF-01); the breaker short-circuits during an outage so timeouts don't pile up (D2-05) |
| T-02-10 | Elevation/Bypass | fail-open under induced outage | accept | fail-open is the INTENDED, configurable default (D2-04 — a limiter must not take down the API when its store is down); the trade-off is documented and fail-closed is available; DESIGN.md (Phase 4) records the rationale |
| T-02-11 | DoS (unhandled rejection) | RedisStore error path | mitigate | every op `catch`es all errors and resolves through `degraded()` — RedisStore never rejects (DEF-02), so a store error cannot crash the caller |
| T-02-SC | Tampering | ioredis/testcontainers installs | mitigate | install gated by the blocking human-verify checkpoint in plan 02-02; no postinstall scripts (02-RESEARCH.md audit) |
</threat_model>
