<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller → limiter.consume | client-controlled `key` and `cost` cross into the core |
| limiter → Store op | `now`, `cost`, `cfg` cross into the (soon Redis-backed) op |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | RedisStoreConfig at construction | mitigate | `assertPolicy`/`assertPrefix`/`assertPositiveConfig` reject garbage config (bad policy enum, empty prefix, non-positive timeout/breaker) before any op runs |
| T-02-02 | Denial of Service | MemoryStore async migration | mitigate | Preserve event-loop atomicity (no `await` in the critical section); concurrency guard test pins exactly-`limit` admission so a torn read-modify-write (over-admission, the highest-severity limiter bug) is caught |
| T-02-03 | Information Disclosure | opaque `key` | accept | core never parses/logs `key` (CORE-05); no PII handling added here |
</threat_model>
