    Create `rate-limiter/src/store/memory.ts` exporting a `MemoryStore` class implementing `Store` from `../types.js`. Hold per-key state in a `Map<string, AlgoState>` (separate internal state shapes per algorithm; a `TBState = { tokens: number; lastRefill: number }`, a fixed-window counter-per-window-bucket state, and a sliding-window prev/curr-counter state). Each op is the 5-step atomic critical section with NO `await` inside (event-loop atomicity — STOR-01/Pitfall 2): (1) load state for key or init defaults, (2) apply algorithm math with the passed `now`, (3) decide all-or-nothing (D-01), (4) write new state ONLY if allowed so rejection leaves state byte-identical (D-01), (5) return the `OpTuple`.

    `tokenBucket(key, cfg: TBConfig, cost, now)`: lazy refill `refilled = min(capacity, tokens + (elapsed/intervalMs)*refillPerInterval)` with `elapsed = max(0, now - lastRefill)` recomputed from the integer `now` each call (never accumulate fractional ms — Pitfall 3); `allowed = cost <= refilled ? 1 : 0` (D-01; `cost > capacity` returns 0 without throwing — D-02); `tokensAfter = allowed ? refilled - cost : refilled`; `remaining = Math.floor(tokensAfter)` (D-04); `resetMs = Math.ceil((capacity - tokensAfter)/refillPerInterval * intervalMs)` (D-05); `retryAfterMs = allowed ? 0 : Math.ceil(max(0, cost - refilled)/refillPerInterval * intervalMs)` (D-03); persist `{ tokens: tokensAfter, lastRefill: now }`.

    `slidingWindow(key, cfg: WindowConfig, cost, now)`: maintain prev-window and curr-window counts keyed by `floor(now/windowMs)`; on window roll, shift curr→prev and reset curr. `overlapFraction = (windowMs - elapsedInCurrent)/windowMs`; `estimate = curr + prev*overlapFraction`; admit when `Math.floor(estimate) + cost <= limit` (D-13). `remaining = max(0, limit - (Math.floor(estimate) + (allowed?cost:0)))`. `resetMs`/`retryAfterMs` are integer ms via `Math.ceil` (time until the window rolls / enough overlap decays to admit). On reject, do NOT increment curr (D-01). Encode the math so the Xu Ch.4 example (limit=7, prev=5, curr=3, 50% in → floor(5.5)=5, 5+1=6<=7 → admit, remaining=1 — D-14) holds; that exact case is asserted by the test plan.

    `fixedWindow(key, cfg: WindowConfig, cost, now)`: counter per `windowMs` bucket keyed by `floor(now/windowMs)`; reset count to 0 when the bucket index changes; admit while `count + cost <= limit`; on admit increment, on reject leave count unchanged (D-01). Do NOT add any smoothing — the 2×-at-boundary burst is REQUIRED behavior (ALGO-03/Pitfall 4). `remaining = max(0, limit - count)` after the decision; `resetMs = Math.ceil(time until next bucket boundary)`; `retryAfterMs = allowed ? 0 : Math.ceil(time until next bucket boundary)`.

    Rounding contract — PIN IN CODE COMMENTS on every outgoing duration and on `remaining`: `Math.floor` for `remaining`, `Math.ceil` for `resetMs` and `retryAfterMs`, each commented with its Lua equivalent (e.g. `// CEIL: matches Lua math.ceil(...)`) because Phase-2 conformance (TEST-02) copies these bit-for-bit. Never let fractional ms cross the op boundary (D-09); fractional token counts live only inside `TBState`. Optimize for clarity over cleverness — this op is the reference impl (APOSD deep module). Add no mutex/lock and no generic get/set (Out-of-Scope slop traps).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller → limiter config | construction-time numeric config (capacity/limit/windowMs/intervalMs/refillPerInterval) is untrusted input |
| caller → consume(key, cost) | per-request `cost` and opaque `key` cross into the atomic op |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-06 | Tampering / DoS | non-positive/NaN/Infinity config causing incorrect admit-reject or unbounded loops | mitigate | Each limiter constructor validates config and throws on non-positive/NaN/non-finite values before any op runs (construction-time guard) |
| T-01-07 | Elevation (the core security property) | over-admission via a torn read-modify-write in a store op | mitigate | Each op is one synchronous critical section with NO `await` (event-loop atomicity); Task 1 verify gate fails if `await` appears inside an op; the exact-limit guard is proven by the concurrency test in plan 01-04 |
| T-01-08 | DoS | unbounded key-space growth in the MemoryStore `Map` (one entry per distinct key) as a memory-exhaustion vector | accept | Documented and bounded to this in-memory reference store; eviction/TTL is intentionally deferred to the Redis store (Phase 2 sets key TTL inside Lua). For a single-node reference impl this is an accepted, noted limitation — surface it in DESIGN.md (Phase 4) |
| T-01-09 | Tampering | `cost > capacity` throwing or partially draining state | mitigate | D-01/D-02: ops are total — `cost > capacity` returns `allowed:false` without consuming; rejection leaves state byte-identical (no partial drain) |
</threat_model>
