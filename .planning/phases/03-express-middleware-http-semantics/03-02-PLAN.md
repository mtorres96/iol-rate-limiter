<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client → Express middleware | Untrusted request crosses here; the middleware extracts a client key and may receive a forged `X-Forwarded-For`. |
| Middleware → `RateLimiter.consume` | An (in-memory or alternative) limiter may reject; the rejection must not crash the request. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-02 | Spoofing | client-key extraction (req.ip / forged X-Forwarded-For) | mitigate | D3-02: rely on Express `trust proxy` to populate `req.ip`; the middleware NEVER parses `X-Forwarded-For` itself (no XFF parsing code exists in middleware.ts). Documented as a deployment note. |
| T-03-03 | Tampering | empty/forged client key | mitigate | D3-03: empty key admits AND logs via `DegradedLogger` (operator visibility); key is treated opaquely and never interpolated into a query. |
| T-03-04 | Denial of Service | unhandled rejection from `consume()` crashing the request | mitigate | D3-07/HTTP-04: `try/catch` around `await consume()` resolves through the middleware-owned fail-open/closed policy; the catch never rethrows and never leaks to the Express error handler. |
| T-03-05 | Tampering | header injection via key/policy values | mitigate | Header values derive only from numeric `Decision` fields and the fixed `"default"` policy name; no user-controlled string is interpolated into a header. |
</threat_model>
